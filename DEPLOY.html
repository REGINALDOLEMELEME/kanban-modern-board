2 Pré-requisitos e backup

Checklist antes de começar

Backup do banco

No phpMyAdmin → seleciona o banco kanban_db → aba Exportar → método "Rápido" → formato SQL → Executar. Guarda o arquivo como kanban_db_backup_2026-04-20.sql.

Via CLI:

mysqldump -u USER -p kanban_db > kanban_db_backup_2026-04-20.sql

3 Migração do banco

!
Ordem importa. Rode o SQL antes de subir o PHP. Se subir o PHP primeiro, as rotas /logs e /templates vão retornar 500 até as tabelas existirem.
  1. Abre o phpMyAdmin e seleciona kanban_db na lateral esquerda.
  2. Clica na aba SQL.
  3. Cola o conteúdo do arquivo backend2/migrations/2026-04-20_templates_and_logs.sql (ou usa Importar e envia o arquivo).
  4. Clica em Executar. Deve aparecer "A sua consulta foi executada com êxito".
i
O script é idempotente (usa CREATE TABLE IF NOT EXISTS e INSERT IGNORE). Rodar duas vezes não quebra nada nem duplica dados.

Verificação rápida

Executa no SQL do phpMyAdmin para confirmar:

SHOW TABLES LIKE 'card_templates';
SHOW TABLES LIKE 'change_logs';
SELECT COUNT(*) AS modelos FROM card_templates;

Esperado: as duas tabelas aparecendo e modelos = 3 (ou mais, se você já tinha alguma).

!
Ordem recomendada: backend primeiro, frontend depois. Assim um usuário que abre o navegador no meio do deploy não pega HTML novo apontando para rotas que ainda não existem.

4.1 · Backend (1 arquivo)

  • Substituir: backend2/api/index.php

4.2 · Frontend (3 arquivos)

  • Substituir: frontend/index.html
  • Substituir: frontend/script.js
  • Substituir: frontend/style.css

⚠ Arquivos que NÃO devem ser enviados

  • backend2/config.php — o do servidor já tem as credenciais de produção. Se subir o do repo, a app para imediatamente.
  • backend2/database_dump.sql — é a seed de desenvolvimento. Vai derrubar e recriar o banco inteiro.
i
Antes de sobrescrever cada arquivo no servidor, renomeie o atual para nome-do-arquivo.old. Isso facilita o rollback caso algo dê errado.

5 Smoke test pós-deploy

Abra a URL de produção em uma janela anônima (Ctrl+Shift+N) para evitar cache antigo. Percorra cada item:

Monitorar erros na primeira hora

Deixe aberto o error log do PHP (cPanel → "Errors" ou tail -f error_log). Qualquer 500 imprevisto aparece lá com o prefixo [kanban] — o código novo escreve toda exceção não tratada.

i
Modo debug temporário: se aparecerem 500s misteriosos, ative APP_DEBUG=1 no servidor (no .htaccess: SetEnv APP_DEBUG 1). Nesse modo a API passa a devolver message/file/line no JSON. Desligue depois — essas informações não devem ficar expostas em produção.

6 Notas importantes

6.1 · CORS restringido

A versão nova não usa mais Access-Control-Allow-Origin: *. Ela mantém uma allow-list fixa (http://127.0.0.1:8787, http://localhost:8787) mais o que estiver na env var KANBAN_ALLOWED_ORIGINS.

  • Se UI e API estão no mesmo domínio (caso normal em produção): o browser não envia o header Origin em GETs, o CORS não é aplicado, não precisa configurar nada.
  • Se UI e API ficam em domínios diferentes: configura no servidor web do backend:
# .htaccess (ou equivalente em Nginx/Caddy)
SetEnv KANBAN_ALLOWED_ORIGINS "https://app.empresa.com"

# múltiplos separados por vírgula
SetEnv KANBAN_ALLOWED_ORIGINS "https://app.empresa.com,https://admin.empresa.com"

6.2 · URL da API resolvida no cliente

O script.js monta a URL da API a partir de window.location:

`${origin}${appBase}backend2/api/index.php?route=...`

Onde appBase é o pedaço do caminho antes de /frontend/. Isso significa que a estrutura de pastas no servidor precisa ser a mesma do repo (pasta frontend/ e pasta backend2/ irmãs dentro da raiz do site).

Para conferir, DevTools → aba Network → clica em criar card → a request deve bater em https://seu-dominio/.../backend2/api/index.php?route=/cards.

6.3 · Browser cache

Se algum usuário relatar "nada mudou", peça para ele dar Ctrl+F5. O ?v=20260420-3 no index.html já força, mas alguns proxies corporativos teimam.

6.4 · Compatibilidade com dados existentes

  • Todos os cards antigos continuam funcionando — as mudanças de schema são aditivas.
  • order_index das cards antigas não precisa ser remapeado. Qualquer colisão pré-existente se resolve naturalmente na próxima reordenação por drag.
  • A tabela change_logs preserva todo o histórico existente. Se já foi criada automaticamente antes, o CREATE TABLE IF NOT EXISTS não a toca.

7 Plano de rollback

!
Se algo grave aparecer nos primeiros minutos, reverta antes de debugar. O debug vem depois, com calma.
  1. Renomeie os 4 arquivos substituídos para .new e renomeie os .old de volta aos nomes originais. O app volta para a versão anterior em segundos.
  2. Se o banco ficou em estado estranho (raro — o SQL só adiciona tabelas novas): 
    DROP TABLE IF EXISTS card_templates;
-- NÃO dropar change_logs se ela já tinha dados antigos.
  3. Em último caso, restaure o backup do passo 2: 
    mysql -u USER -p kanban_db < kanban_db_backup_2026-04-20.sql
  4. Peça aos usuários Ctrl+F5 para voltar a ver a versão antiga.
